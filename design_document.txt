3. Core Functional Requirements
3.1 Tournament & Match Data

Integrate with a soccer data API such as Football-Data.org or API-Football.

The system must:

sync tournaments, teams, groups, stages, and matches
store kickoff time, venue, status, and final result
cache external API responses to avoid rate limit issues
support manual admin refresh
support scheduled background syncs
support manual admin override of match data if the provider is wrong or delayed

Design the code so the external API provider is abstracted behind a service layer and can be replaced later.

3.2 Predictions

Users can submit one prediction per match:

home team score
away team score

Rules:

predictions are editable only until the match enters lock-out
after lock-out, predictions become read-only
once a match is completed and results are confirmed, points are calculated automatically
recalculation must be idempotent
3.3 Lock-out Logic

Lock-out must be configurable by admins.

Requirements:

configurable as hours before kickoff
default example: 1 hour before kickoff
store the resolved lock_at timestamp per match so rule changes do not retroactively affect already published match windows unless an admin explicitly reprocesses them
all time handling must use UTC internally
UI may display times in the user’s local timezone
server-side validation must enforce lock-out even if the client is bypassed
3.4 Scoring System

Implement the default scoring model:

3 points for exact score
1 point for correct outcome only
0 points otherwise

Outcome means:

home win
draw
away win

Also make the scoring engine extensible to support future variants such as:

weighted points for underdog picks
bonus points for knockout matches
special round multipliers

Do not enable weighted scoring by default, but structure the code so it can be turned on later.

3.5 Leaderboards

Support:

global leaderboard
leaderboard by round/stage
optional friends/challenge leaderboard later

Include tie-breakers in this exact order:

highest exact-score hits
highest correct-outcome hits
earliest average prediction submission time before lock
alphabetical username as final stable tie-breaker

Redis should cache leaderboard views for performance, but PostgreSQL remains the source of truth.

3.6 Social & Engagement
The Lounge

Each match has a comment feed.

Requirements:

authenticated users can post comments
users can edit/delete only their own comments within a short grace period, e.g. 5 minutes
admins can moderate/delete any comment
comments should support pagination
basic anti-abuse protections:
rate limit posting
max comment length
optional banned-word filter
real-time updates preferred via polling or websocket/SSE, but polling is acceptable if simpler
User Challenges / Duals

Users can challenge each other for a specific match.

Requirements:

one user initiates a challenge
target user must accept or decline
both users submit predictions normally
once the match completes, winner of the duel is determined by prediction points
if tied, mark duel as draw
include a token/badge/streak system only for engagement, with no real money or gambling language
allow users to see duel history

Comments & Moderation

Leaderboards & Winners
The system must have one main leaderboard

Prediction Editing
Users may edit their predictions multiple times until the match lock deadline

Admin Capabilities
Admins may support users through management tools

Flash Challenges

Notifications
